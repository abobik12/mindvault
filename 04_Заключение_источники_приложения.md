# ЗАКЛЮЧЕНИЕ

В ходе выполнения выпускной квалификационной работы была достигнута поставленная цель: разработано кроссплатформенное веб-приложение "Личный секретарь" с использованием ИИ. Внутри работы приложение имеет название MindVault. Под кроссплатформенностью в рамках проекта понимается возможность работы через браузер на настольных и мобильных устройствах без создания отдельного нативного мобильного клиента.

Разработанное приложение предназначено для хранения, структурирования и поиска персональной цифровой информации пользователя. В состав реализованных функций входят регистрация и вход пользователя, работа с заметками, файлами, папками, напоминаниями и списками, а также чат с ИИ-ассистентом. Ассистент использует сохраненный пользовательский контекст при формировании ответа и может выполнять прикладные действия через серверную обработку команд.

На первом этапе работы был выполнен анализ предметной области. Было рассмотрено, как пользовательская информация обычно распределяется между заметками, файлами, списками задач, напоминаниями, мессенджерами и ИИ-чатами. Такой анализ показал, что основная сложность состоит не только в сохранении данных, но и в повторном обращении к ним. Если заметки, файлы и задачи хранятся в разных сервисах, пользователь вынужден самостоятельно помнить, где находится конкретный материал и в каком виде он был сохранен.

Также были рассмотрены существующие решения и аналоги: облачные хранилища, сервисы заметок, планировщики задач, мессенджеры с сохраненными сообщениями, локальные базы знаний и ИИ-рабочие пространства. Каждый из этих классов решений закрывает отдельную часть задачи, но не дает единой модели "личного секретаря", в которой заметки, файлы, списки, напоминания и ИИ-ассистент связаны с одной рабочей областью пользователя. На основе этого были сформулированы функциональные и нефункциональные требования к MindVault.

В работе был выбран стек технологий, соответствующий масштабу учебного проекта и фактической реализации. Для frontend-части использованы React, TypeScript и Vite. Клиентская часть построена как одностраничное приложение с защищенными маршрутами и отдельными страницами для чата, заметок, файлов, списков, напоминаний, папок и настроек. Для backend-части выбран Express с TypeScript. Сервер предоставляет REST API, выполняет авторизацию, обрабатывает CRUD-операции, загрузку файлов, поиск, статистику, сообщения чата и команды ассистента. Для хранения данных используется PostgreSQL, а схема описана через Drizzle ORM.

На втором этапе была спроектирована архитектура приложения. MindVault включает браузерный frontend, серверный API, базу данных PostgreSQL и внешний OpenAI/OpenRouter-совместимый API. В проекте предусмотрена работа через Docker Compose, где отдельно запускаются сервисы `web`, `api` и `db`. База данных спроектирована вокруг таблиц `users`, `folders`, `items`, `conversations` и `messages`. Центральной сущностью является таблица `items`, через которую хранятся заметки, файлы, напоминания и списки.

В ходе проектирования были определены пользовательские сценарии. Пользователь может зарегистрироваться, войти в систему, создать заметку, загрузить файл, создать папку, добавить напоминание, сформировать список, выполнить поиск, задать вопрос ассистенту и обратиться к сохраненному контексту. Для чата отдельно предусмотрена обработка пользовательских сообщений: часть запросов рассматривается как обычный вопрос к ассистенту, а часть как прикладная команда, например создание заметки или списка.

Практическая реализация проекта выполнена в виде TypeScript-монорепозитория. Frontend расположен в `artifacts/mindvault`, backend - в `artifacts/api-server`, общие библиотеки - в `lib`. В проекте также присутствуют сгенерированные API-клиенты, Zod-схемы, OpenAPI-описание, конфигурация Docker и файл `TESTING.md` с ручными сценариями проверки.

Авторизация реализована с использованием JWT. На frontend защищенные маршруты проверяют наличие токена в `localStorage`, а backend проверяет токен в middleware `requireAuth`. Пароли пользователей не хранятся в открытом виде: при регистрации используется хеширование через `bcryptjs`. При этом текущую авторизацию следует рассматривать как базовый вариант для учебного проекта, поскольку refresh token, расширенное управление сессиями и end-to-end encryption не реализованы.

Работа с данными пользователя реализована через серверные маршруты и таблицы PostgreSQL. Заметки хранятся как объекты типа `note`, файлы - как `file`, напоминания - как `reminder`, списки - как `list`. Папки позволяют группировать объекты, а при удалении пользовательской папки связанные объекты не удаляются, а остаются без привязки к папке. Это снижает риск случайной потери данных.

Работа с файлами реализована через endpoint загрузки. Файл передается на сервер вместе с именем, MIME-типом, размером и содержимым в формате base64. Для текстовых файлов, PDF и DOCX выполняется попытка извлечения текста. Извлеченный текст сохраняется и может использоваться ассистентом при формировании ответа. Если формат не поддерживается или текст извлечь не удалось, файл сохраняется с метаданными, но система не должна выдумывать его содержимое. OCR для изображений и сканов в текущей версии не реализован.

ИИ-ассистент реализован через OpenAI/OpenRouter-совместимый API. Backend формирует контекст пользователя на основе сохраненных заметок, файлов, списков, напоминаний и папок. При обращении к ассистенту этот контекст используется для ответа. Отдельно реализована серверная обработка команд: создание заметок, списков и напоминаний, поиск объектов, работа с папками и подтверждение потенциально опасных действий. В текущей версии не следует утверждать, что все намерения пользователя классифицируются большой языковой моделью: основной механизм прикладных команд реализован серверной логикой и правилами.

В проекте предусмотрен fallback при недоступности внешнего AI API. Если ключ провайдера не задан или внешний сервис недоступен, основные функции приложения продолжают работать, а чат возвращает ответ о недоступности AI и, при возможности, использует локальный контекст. Это делает приложение менее зависимым от внешнего сервиса в части базовой работы с заметками, файлами, списками и напоминаниями.

Тестирование показало, что автоматические проверки ключевой серверной логики выполняются успешно. Были запущены тесты command-parser backend: 5 тестов из 5 завершились успешно. Также были выполнены TypeScript-проверки backend и frontend, ошибок типов не выявлено. Ручные сценарии проверки описаны отдельно и должны быть дополнены финальными скриншотами перед сдачей работы.

Практическая значимость результата состоит в том, что MindVault может использоваться как персональное рабочее пространство для хранения заметок, файлов, напоминаний и списков, а также для обращения к сохраненному контексту через ИИ-ассистента. Пользователь получает не только отдельные разделы интерфейса, но и единый чатовый способ взаимодействия с личной информацией.

При этом текущая версия имеет ограничения, которые важно честно отразить в работе. В коде не реализованы push-уведомления, OCR, semantic/vector search, production-деплой через HTTPS/nginx, совместная работа нескольких пользователей, end-to-end encryption, интеграции с календарями и мессенджерами, платежи и роли администратора. Эти направления не следует описывать как готовые функции. Их можно рассматривать как перспективы развития.

К возможным направлениям дальнейшего развития относятся добавление Web Push-уведомлений по напоминаниям, внедрение семантического поиска с использованием embeddings, подключение OCR для изображений и сканов, перенос файлов из PostgreSQL в объектное хранилище, подготовка production-конфигурации с HTTPS, развитие PWA-возможностей, расширение модели безопасности и добавление совместной работы через рабочие пространства.

Итогом работы является спроектированное и реализованное веб-приложение MindVault, включающее frontend, backend, базу данных, AI-интеграцию, Docker Compose-запуск и набор проверок. Полученный результат соответствует поставленной цели и может быть использован как основа для дальнейшего развития персональной информационной системы с ИИ-ассистентом.

## Фрагмент 2. Middleware `requireAuth`

Файл: `artifacts/api-server/src/middlewares/auth.ts`.

Фрагмент подтверждает, что backend проверяет JWT-токен из заголовка `Authorization` и добавляет данные пользователя в `req.auth`.

```ts
export function generateToken(payload: AuthPayload): string {
  return jwt.sign(payload, JWT_SECRET, { expiresIn: "30d" });
}

export function verifyToken(token: string): AuthPayload | null {
  try {
    return jwt.verify(token, JWT_SECRET) as AuthPayload;
  } catch {
    return null;
  }
}

export function requireAuth(req: Request, res: Response, next: NextFunction): void {
  const authHeader = req.headers.authorization;

  if (!authHeader || !authHeader.startsWith("Bearer ")) {
    res.status(401).json({ error: "Authorization required" });
    return;
  }

  const token = authHeader.slice(7);
  const payload = verifyToken(token);

  if (!payload) {
    res.status(401).json({ error: "Invalid or expired token" });
    return;
  }

  req.auth = payload;
  next();
}
```

## Фрагмент 4. Извлечение текста из загруженного файла

Файл: `artifacts/api-server/src/lib/file-extraction.ts`.

Фрагмент подтверждает, что система пытается извлекать текст из текстовых файлов, PDF и DOCX, а для неподдерживаемых форматов сохраняет статус `unsupported`.

```ts
export async function extractTextFromUpload(
  filename: string,
  mimeType: string,
  fileData: string,
): Promise<FileExtractionResult> {
  const buffer = Buffer.from(fileData, "base64");

  try {
    if (isPlainTextUpload(filename, mimeType)) {
      const text = validateTextLike(buffer.toString("utf8"));
      return text
        ? { text, status: "completed", summary: extractionSummary("completed", "plain text extracted") }
        : { text: null, status: "failed", summary: extractionSummary("failed", "plain text could not be decoded safely") };
    }

    if (isPdfUpload(filename, mimeType)) {
      const parsed = await loadPdfParser()(buffer);
      const text = cleanExtractedText(parsed.text);
      return text
        ? { text, status: "completed", summary: extractionSummary("completed", "pdf text extracted") }
        : { text: null, status: "failed", summary: extractionSummary("failed", "pdf parser returned no text") };
    }

    if (isDocxUpload(filename, mimeType)) {
      const parsed = await loadMammoth().extractRawText({ buffer });
      const text = cleanExtractedText(parsed.value);
      return text
        ? { text, status: "completed", summary: extractionSummary("completed", "docx text extracted") }
        : { text: null, status: "failed", summary: extractionSummary("failed", "docx parser returned no text") };
    }

    return { text: null, status: "unsupported", summary: extractionSummary("unsupported", "format is metadata-only") };
  } catch (err) {
    const message = err instanceof Error ? err.message.slice(0, 180) : "unknown extraction error";
    return { text: null, status: "failed", summary: extractionSummary("failed", message) };
  }
}
```

```ts
  await db
    .update(itemsTable)
    .set({ content: extraction.text, summary: extraction.summary })
    .where(and(eq(itemsTable.id, item.id), eq(itemsTable.userId, userId), eq(itemsTable.type, "file")));
}
```

## Фрагмент 7. Fallback при ошибке AI API

Файл: `artifacts/api-server/src/routes/gemini.ts`.

Фрагмент подтверждает, что при ошибке внешнего AI API сервер сохраняет работоспособность чата и формирует fallback-ответ.

```ts
try {
  // Отправка запроса к OpenAI/OpenRouter-compatible API
  // и потоковая передача ответа пользователю.
} catch (err) {
  logger.error({ err }, "[gemini] AI response failed");

  const fallbackResponse = buildOfflineAssistantReply(
    userContent,
    err instanceof Error ? err.message : "unknown error",
    context,
  );

  await db.insert(messages).values({
    conversationId,
    role: "assistant",
    content: fallbackResponse,
    metadata: { mode: "offline_fallback" },
  });
}
```

# ПРИЛОЖЕНИЕ 4. Инструкция по запуску приложения через Docker

## Предварительные требования

Для запуска приложения через Docker необходимо установить Docker и Docker Compose. Для локального режима разработки без контейнеров также потребуются Node.js и `pnpm`, однако при запуске через `docker compose` основные сервисы запускаются внутри контейнеров.

Перед запуском рекомендуется проверить наличие файла `.env` или переменных окружения. В текущем `docker-compose.yml` часть переменных имеет значения по умолчанию, но для полноценной работы AI-ассистента необходимо задать ключ одного из AI-провайдеров.

## Основные команды

1. Собрать и запустить контейнеры:

```bash
docker compose up -d --build
```

2. Проверить состояние контейнеров:

```bash
docker compose ps
```

3. Проверить логи backend:

```bash
docker compose logs api
```

4. Проверить логи frontend:

```bash
docker compose logs web
```

5. Открыть frontend-приложение:

```text
http://localhost:18174
```

6. Проверить доступность backend:

```text
http://localhost:8080/api/healthz
```

7. Остановить контейнеры:

```bash
docker compose down
```

## Сервисы Docker Compose

В файле `docker-compose.yml` описаны три сервиса:

- `db` - база данных PostgreSQL 16, доступная на порту `5432`;
- `api` - backend-сервер Express, доступный на порту `8080`;
- `web` - frontend-приложение Vite, доступное на порту `18174`.

Сервис `api` ожидает готовности базы данных, после чего применяет схему командой:

```bash
pnpm --filter @workspace/db run push
```

Затем backend запускается командой:

```bash
pnpm --filter @workspace/api-server run dev
```

Сервис `web` запускает frontend:

```bash
pnpm --filter @workspace/mindvault run dev
```

## Переменные окружения

В текущей конфигурации используются следующие переменные:

- `PORT` - порт backend-сервера внутри контейнера;
- `NODE_ENV` - режим работы приложения;
- `SESSION_SECRET` - секрет для подписи JWT;
- `DATABASE_URL` - строка подключения к PostgreSQL;
- `OPENAI_BASE_URL` - базовый URL OpenAI-совместимого API;
- `OPENAI_MODEL` - модель OpenAI по умолчанию;
- `OPENAI_API_KEY` - ключ OpenAI API;
- `OPENROUTER_BASE_URL` - базовый URL OpenRouter API;
- `OPENROUTER_MODEL` - модель OpenRouter по умолчанию;
- `OPENROUTER_API_KEY` - ключ OpenRouter API;
- `OPENROUTER_SITE_URL` - URL приложения для OpenRouter;
- `OPENROUTER_APP_NAME` - название приложения для OpenRouter;
- `ASSISTANT_SYSTEM_PROMPT` - необязательное системное сообщение ассистента;
- `BASE_PATH` - базовый путь frontend-приложения;
- `API_PROXY_TARGET` - адрес backend для проксирования API-запросов из Vite.

Если `OPENAI_API_KEY` и `OPENROUTER_API_KEY` не заданы, основные функции приложения продолжают работать: регистрация, вход, заметки, файлы, папки, напоминания, списки и поиск не требуют внешнего AI API. Чат в этом случае не сможет получить ответ от внешней модели и должен вернуть fallback-сообщение.

Текущий Docker Compose используется для локального и демонстрационного запуска. Он не описывает production-деплой с HTTPS, nginx, доменным именем и отдельной инфраструктурой мониторинга.
