# ВВЕДЕНИЕ

Пользовательская информация редко хранится в одном месте. Файлы находятся в облачном хранилище или на локальном диске, короткие мысли и ссылки попадают в заметки, задачи ведутся в отдельном планировщике, а часть материалов остается в переписках. Такой способ работы привычен, но он создает несколько практических проблем: данные трудно найти повторно, между ними нет единой структуры, а пользователь вынужден помнить, в каком сервисе был сохранен конкретный фрагмент информации.

Отдельную задачу составляет работа с ИИ-ассистентом. Обычный чат с ИИ помогает сформулировать ответ, подготовить текст или объяснить материал, но сам по себе он не является долговременным персональным хранилищем. Если ассистент не связан с заметками, файлами, списками и напоминаниями пользователя, то он не может устойчиво использовать личный контекст. Пользователь снова вручную прикрепляет документы, пересказывает ранее сохраненные сведения или ищет нужную информацию до обращения к ассистенту.

Для решения этой задачи в работе разрабатывается приложение, которое объединяет хранение персональной информации и чатовый способ взаимодействия. Чат используется для быстрого ввода: пользователь может написать фразу естественным языком, попросить сохранить заметку, найти материал, создать напоминание или задать вопрос по сохраненному контексту. При этом в приложении остаются обычные разделы для просмотра и редактирования данных: заметки, файлы, папки, напоминания и списки. Пользователь получает быстрый чатовый ввод, но не теряет предсказуемую структуру классической информационной системы.

В работе рассматривается разработка кроссплатформенного веб-приложения "Личный секретарь" с использованием ИИ. Внутри работы проект имеет название MindVault. Под кроссплатформенностью в рамках проекта понимается доступность приложения через браузер на настольных и мобильных устройствах без разработки отдельного нативного мобильного клиента. Это соответствует выбранной архитектуре: клиентская часть работает как веб-приложение, серверная часть предоставляет API, а данные хранятся в базе данных PostgreSQL.

Объект исследования - процесс хранения, структурирования и поиска персональной цифровой информации пользователя.

Предмет исследования - методы и средства разработки кроссплатформенного веб-приложения с ИИ-ассистентом для работы с заметками, файлами, напоминаниями и списками.

Цель работы - разработать кроссплатформенное веб-приложение "Личный секретарь" с использованием ИИ, обеспечивающее хранение, структурирование и поиск персональной информации пользователя через единый чат-интерфейс.

Для достижения цели необходимо решить следующие задачи:

1. Провести анализ предметной области и существующих решений.
2. Сформулировать функциональные и нефункциональные требования.
3. Обосновать выбор инструментов разработки.
4. Спроектировать архитектуру приложения и структуру базы данных.
5. Реализовать frontend, backend и базу данных.
6. Реализовать ИИ-ассистента и механизм работы с пользовательским контекстом.
7. Реализовать работу с заметками, файлами, папками, напоминаниями и списками.
8. Провести тестирование основных сценариев.

В работе применяются следующие методы: анализ предметной области, сравнительный анализ аналогов, формулирование требований, проектирование архитектуры информационной системы, проектирование базы данных, моделирование пользовательских сценариев, прототипирование, разработка и тестирование веб-приложения.

Практическая значимость работы состоит в создании приложения, которое может использоваться как персональное рабочее пространство для хранения заметок, файлов, задач и взаимодействия с ИИ-ассистентом. Пользователь получает единый интерфейс для добавления, поиска и повторного использования информации, а не набор разрозненных сервисов.

Работа состоит из введения, трех глав, заключения, списка использованных источников и приложений. Оформление текстовой части выполняется с учетом требований и методических материалов КГУ [1], [2]. В третьей главе будет описана реализация и тестирование разработанного веб-приложения.

# ГЛАВА 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ И ПОСТАНОВКА ТРЕБОВАНИЙ К СИСТЕМЕ

## 1.1. Предметная область и актуальность задачи

Персональная цифровая информация включает данные, которые пользователь создает, получает и использует в повседневной учебной, рабочей и личной деятельности. К ней относятся текстовые заметки, документы, изображения, ссылки, идеи, списки дел, напоминания, учебные материалы, рабочие файлы и фрагменты переписки. В отличие от корпоративных информационных систем, где структура данных часто задается заранее, персональные данные появляются нерегулярно и имеют разный формат.

Проблема заключается не только в хранении. Сам факт сохранения файла или заметки еще не означает, что пользователь сможет быстро найти нужный материал позже. Если документ лежит в одной папке, заметка записана в другом приложении, а задача создана в отдельном планировщике, то поиск превращается в последовательную проверку нескольких интерфейсов. При большом количестве материалов это занимает больше времени, чем само сохранение.

На практике пользователь часто выбирает самый быстрый способ фиксации информации. Он отправляет сообщение самому себе в мессенджере, сохраняет файл в загрузки, пишет идею в заметках телефона или оставляет задачу в голове до момента, когда появится время ее оформить. Такой подход удобен в момент ввода, но плохо работает при повторном использовании данных. Через несколько дней становится сложно вспомнить, где именно была сохранена информация и как она называлась.

Персональная информационная система должна решать три связанные задачи. Во-первых, она должна принимать данные разных типов: текст, файлы, списки и напоминания. Во-вторых, она должна сохранять структуру: папки, типы объектов, даты, статусы и связи с пользователем. В-третьих, она должна помогать находить и использовать сохраненную информацию. Если одна из этих частей отсутствует, система становится либо простым хранилищем, либо обычным чатом, либо набором отдельных разделов без общего сценария.

Чатовый интерфейс удобен для быстрого добавления информации, потому что пользователь может не думать о форме заранее. Например, фраза "напомни завтра отправить отчет" ближе к естественному поведению, чем открытие формы, выбор даты, времени, папки и сохранение записи. Однако чат сам по себе недостаточен: он должен быть связан с базой данных и проверяемыми действиями. Если сообщение похоже на команду, система должна корректно определить намерение, выполнить действие и подтвердить результат только после успешной записи данных.

ИИ-ассистент в такой системе рассматривается не как замена интерфейса, а как дополнительный слой работы с контекстом. Он может отвечать на вопросы по сохраненным материалам, кратко пересказывать информацию, помогать ориентироваться в файлах и заметках. При этом границы MVP должны быть описаны отдельно: ассистент не получает функций, которые не подтверждены кодом проекта.

MindVault рассматривается как веб-приложение "личный секретарь", потому что его основная роль состоит в поддержке пользователя при работе с личной информацией. Система должна хранить разные типы данных, давать доступ к ним через разделы интерфейса и позволять обращаться к ним через чат с ИИ-ассистентом. Такой подход уменьшает зависимость от разрозненных инструментов и делает персональную информацию более пригодной для повторного использования.

## 1.2. Обзор существующих решений и аналогов

Для определения требований к MindVault необходимо рассмотреть решения, которые частично закрывают похожие задачи. Аналоги можно условно разделить на несколько групп: облачные хранилища, сервисы заметок, планировщики задач, мессенджеры с сохраненными сообщениями, локальные базы знаний и ИИ-рабочие пространства. Ниже рассмотрены решения, которые часто используются для хранения персональной информации.

Google Drive относится к облачным хранилищам. Сервис предназначен для хранения файлов, их синхронизации и доступа с разных устройств. Пользователь может создавать папки, загружать документы, открывать файлы в связанных сервисах и предоставлять доступ другим пользователям. Преимущество Google Drive заключается в надежной работе с файлами и удобной кроссплатформенности. Ограничение относительно задачи MindVault состоит в том, что сервис ориентирован прежде всего на файлы. Заметки, списки, напоминания и чатовый сценарий работы с ИИ не являются его основной моделью.

Notion представляет собой рабочее пространство для заметок, баз данных, страниц и совместной работы. Он хорошо подходит для структурирования информации и ведения личных или командных баз знаний. Пользователь может создавать страницы, таблицы, шаблоны и связи между объектами. Сильная сторона Notion - гибкость структуры. В рассматриваемой задаче это одновременно ограничение: часть структуры пользователь настраивает вручную. Для быстрого сохранения разнородной информации через единый чат-интерфейс Notion подходит не во всех сценариях.

Evernote относится к сервисам заметок, рассчитанным на сохранение текстов, изображений, документов и веб-фрагментов. Он удобен для ведения личного архива и поиска по заметкам. Важное преимущество Evernote - ориентация на сбор материалов. При этом сервис остается в первую очередь приложением для заметок. Задачи, файлы, напоминания и ИИ-взаимодействие существуют не как единая модель "личного секретаря", а как отдельные возможности вокруг заметок.

Telegram "Избранное" часто используется как простое персональное хранилище. Пользователь может отправить туда текст, файл, ссылку, изображение или голосовое сообщение. Основное преимущество такого подхода - скорость ввода и привычный интерфейс мессенджера. Недостаток заключается в слабой структуре: сообщения идут лентой, папки и типы объектов выражены ограниченно, а повторный поиск старых материалов зависит от текста сообщения и памяти пользователя.

Todoist и Microsoft To Do представляют класс планировщиков задач. Они позволяют создавать задачи, сроки, списки, проекты и напоминания. Эти сервисы хорошо решают задачу управления делами, но не предназначены для полноценного хранения файлов, заметок и пользовательского контекста для ИИ-ассистента. Для MindVault такие решения важны как ориентир по работе со списками и напоминаниями, но они не закрывают всю предметную область.

AI workspace-инструменты и проекты в ИИ-чатах позволяют вести диалог с языковой моделью, прикреплять файлы и организовывать работу вокруг отдельных тем. Их преимущество - сильная сторона ИИ: объяснение, генерация текста, анализ прикрепленного контекста. Ограничение состоит в том, что такие инструменты не всегда являются обычной персональной информационной системой с собственными сущностями "заметка", "файл", "папка", "список" и "напоминание". В рамках MindVault ИИ должен работать не отдельно от данных, а поверх пользовательского хранилища.

Obsidian относится к локальным базам знаний. Он хранит заметки в Markdown-файлах и позволяет строить связи между ними. Пользователь получает контроль над данными и гибкую структуру. Недостаток для рассматриваемой задачи состоит в том, что Obsidian в основном ориентирован на заметки и базу знаний. Работа с напоминаниями, списками, файлами и ИИ-ассистентом требует дополнительных расширений и настройки.

Анализ аналогов показывает, что каждое решение хорошо закрывает свою область: облачные хранилища удобны для файлов, сервисы заметок - для текстов, планировщики - для задач, мессенджеры - для быстрого ввода, а ИИ-инструменты - для диалога и обработки контекста. Задача MindVault состоит не в замене всех этих сервисов, а в разработке учебного MVP, где основные типы персональной информации объединены в одном веб-приложении.

## 1.4. Функциональные и нефункциональные требования

Требования к приложению удобно разделить на функциональные и нефункциональные. Функциональные требования описывают, что должна делать система. Нефункциональные требования задают ограничения к качеству, архитектуре, запуску, безопасности и удобству использования.

Функциональные требования к MindVault сформулированы по методу MoSCoW. Этот подход разделяет требования на группы Must, Should, Could и Won't. Обязательные требования входят в состав MVP, желательные требования улучшают пользовательский сценарий, возможные требования относятся к развитию, а группа Won't фиксирует то, что не входит в текущую версию. ## 1.6. Постановка задачи разработки

На основе анализа предметной области, аналогов и требований необходимо разработать кроссплатформенное веб-приложение MindVault. Приложение должно обеспечивать регистрацию и вход пользователя, хранение заметок, файлов, папок, напоминаний и списков, поиск по сохраненным данным и взаимодействие с ИИ-ассистентом через чат.

Система должна состоять из frontend-части, backend-части и базы данных. Frontend отвечает за пользовательский интерфейс, навигацию, формы, списки объектов и чат. Backend предоставляет API для авторизации, работы с объектами и взаимодействия с AI-провайдером. База данных хранит пользователей, папки, элементы хранилища, диалоги и сообщения.

Отдельной задачей является работа с пользовательским контекстом. ИИ-ассистент должен использовать сохраненные данные пользователя при формировании ответа, но не должен подменять собой базу данных. Создание заметок, списков, напоминаний и других объектов должно выполняться через серверную логику и подтверждаться только после успешной записи. Такой подход снижает риск ситуации, когда ассистент сообщает об успешном действии, которое фактически не было выполнено.

Разработка должна учитывать границы текущей версии. В состав MVP входят браузерный интерфейс, API, база данных, авторизация, основные разделы приложения, чат с ИИ-ассистентом, работа с файлами и Docker-запуск. Не входящие в MVP функции рассматриваются только как перспективы развития и подробно фиксируются в разделе 3.10.

# ГЛАВА 2. ПРОЕКТИРОВАНИЕ ВЕБ-ПРИЛОЖЕНИЯ MINDVAULT

## 2.1. Общая концепция приложения

MindVault проектируется как кроссплатформенное веб-приложение "личный секретарь" для хранения и повторного использования персональной цифровой информации. В рамках работы под кроссплатформенностью понимается доступ через браузер на настольных и мобильных устройствах. Отдельное нативное мобильное приложение в текущую версию не входит.

Основная идея приложения - объединить несколько привычных типов данных в одном рабочем пространстве. Пользователь может хранить заметки, файлы, папки, напоминания и списки, а также взаимодействовать с ИИ-ассистентом через чат. При этом чат не заменяет базу данных и не становится единственным интерфейсом. Он выступает как дополнительный способ ввода команд, поиска информации и получения ответов с учетом сохраненного пользовательского контекста.

В приложении предусмотрены два способа работы. Первый способ - классический интерфейс с отдельными разделами: "Чат", "Заметки", "Списки", "Файлы", "Напоминания", папки и настройки. Пользователь открывает нужный раздел и выполняет действие через форму, карточку или кнопку. Второй способ - чатовый интерфейс, где пользователь может задать вопрос ассистенту или ввести явную команду на создание объекта. Этот вариант удобен для быстрого добавления информации, но при проектировании важно отделять обычный вопрос от действия, изменяющего данные.

MindVault обрабатывает следующие основные типы данных:

- учетные записи пользователей;
- папки пользователя;
- элементы хранилища с типами `note`, `file`, `reminder`, `list`;
- диалоги с ассистентом;
- сообщения в диалогах.

Для пользователя приложение закрывает несколько практических сценариев. Во-первых, оно дает единое место для хранения коротких текстов, файлов, списков и напоминаний. Во-вторых, оно позволяет разделять данные по папкам и искать сохраненные объекты. В-третьих, оно связывает ИИ-ассистента с пользовательским контекстом: при ответе ассистент может учитывать сохраненные заметки, файлы, папки, списки, напоминания и релевантные сообщения.

Границы текущей версии определены кодом проекта. В этом разделе они задаются на уровне пользовательских сценариев: обязательными считаются только те действия, которые поддержаны frontend, backend и схемой базы данных. Неподтвержденные функции не включаются в основной состав MVP и отдельно рассматриваются в разделе 3.10.

| UC-12 | Пользователь | Явная команда создания напоминания | Пользователь вводит команду напоминания с датой или временем; серверная логика разбирает дату и создает объект типа `reminder`, если данных достаточно. | Средний |
| UC-13 | Пользователь | Неоднозначное сообщение | Если сообщение похоже на напоминание или список, но намерение не является достаточно явным, система предлагает варианты действий вместо автоматического создания объекта. | Средний |
| UC-14 | Пользователь | Удаление объекта | Пользователь удаляет заметку, файл, список или напоминание из соответствующего раздела либо через серверную action-логику с подтверждением. | Средний |
| UC-15 | Пользователь | Изменение профиля | Пользователь открывает настройки и изменяет имя или ссылку на аватар; backend обновляет профиль текущего пользователя. | Низкий |

Сценарии UC-11, UC-12 и UC-13 важны для проектирования чатового интерфейса. Ассистент не должен подтверждать сохранение объекта, если соответствующее действие не было выполнено backend-логикой. Поэтому создание и изменение данных проектируются как серверные операции, а не как свободный текстовый ответ модели.

## 2.3. Архитектура системы

Архитектура MindVault разделена на четыре основные части: frontend, backend, базу данных и внешний AI API. Такое разделение подходит для MVP, потому что каждая часть имеет понятную ответственность и может проверяться отдельно.

Frontend-часть расположена в `artifacts/mindvault`. Она реализована на React, TypeScript и Vite [3], [4], [9]. Навигация построена через Wouter. В файле `App.tsx` определены маршруты `/auth`, `/`, `/notes`, `/lists`, `/files`, `/reminders`, `/folders/:id` и `/settings`. Защищенные маршруты проверяют наличие токена `mindvault_token` в `localStorage`. Основной layout содержит боковую навигацию, список папок, переходы к разделам и выход из аккаунта.

Backend-часть расположена в `artifacts/api-server`. Она реализована на Express 5 и TypeScript [5], [9], [14]. В `app.ts` подключены логирование, CORS, JSON parser с лимитом 25 МБ и общий роутер `/api`. Роуты разделены по файлам: `auth.ts`, `folders.ts`, `items.ts`, `gemini.ts`, `health.ts`. Сервер получает порт из переменной окружения `PORT`.

База данных описана в `lib/db/src/schema`. Для работы с PostgreSQL используется Drizzle ORM [6], [10]. В проекте есть таблицы пользователей, папок, элементов хранилища, диалогов и сообщений. Сущность `items` используется как универсальная таблица для заметок, файлов, напоминаний и списков. Такой подход упрощает общую фильтрацию и поиск по объектам, но требует аккуратной работы с полями, которые относятся только к отдельным типам.

Внешний AI API подключается через модуль `lib/integrations-gemini-ai/src/client.ts`. Несмотря на название интеграционного пакета, фактический клиент использует OpenAI/OpenRouter-compatible endpoint `/chat/completions`. Провайдер выбирается по переменным окружения `OPENROUTER_API_KEY` и `OPENAI_API_KEY`. По умолчанию для OpenAI указан `gpt-4.1-mini`, для OpenRouter - `openai/gpt-4o-mini` [7].

Такой вариант проектирования позволяет использовать единые маршруты `/items` для разных типов объектов. Например, заметка и список различаются значением `type` и содержанием `content`, файл дополнительно использует поля `originalFilename`, `mimeType`, `fileSize` и `fileData`, а напоминание использует поле `reminderAt`.

Связь `folders` и `items` спроектирована мягко: при удалении папки связанные элементы не удаляются, а их `folderId` становится `null`. Для системных папок предусмотрено поле `isSystem`, а backend запрещает их изменение и удаление.

Для таблицы `messages` предусмотрено поле `metadata`. Оно используется для хранения дополнительной информации о сообщении: результата action-логики, sources, вложений и других данных, которые не должны быть отдельными таблицами в текущей версии. Такой подход удобен для MVP, но в дальнейшем отдельные типы metadata можно вынести в более строгие сущности.

Frontend отправляет сообщение в backend, backend сохраняет пользовательское сообщение, определяет возможные действия, формирует контекст и обращается к внешнему AI API. Ответ ассистента также сохраняется в таблице `messages`.

Она обрабатывает явные команды, поиск по пользовательским данным, создание заметок, списков, напоминаний, создание и изменение папок, перенос объектов, удаление объектов с подтверждением и очистку истории чата. Это не semantic/vector search, а обычный контекстный подбор по сохраненным данным.

Если возможно, в ответ включается краткая сводка по найденному пользовательскому контексту. Это позволяет не блокировать работу всего приложения из-за ошибки внешнего сервиса.

Работа с файлами проектируется как часть общей модели `items`. Файл является объектом типа `file`. Frontend передает файл в backend через endpoint `/api/items/upload`. В запрос входят имя файла, MIME-тип, размер, base64-представление содержимого и при необходимости `folderId`.

[ВСТАВИТЬ РИС. 6. Схема загрузки и обработки файла]

Рис. 6. Схема загрузки и обработки файла

На рис. 6 нужно показать выбор файла во frontend, преобразование в base64, отправку на `/api/items/upload`, извлечение текста для поддерживаемых форматов, сохранение метаданных и содержимого в таблице `items`, а также дальнейшее использование текста файла в поиске и контексте ассистента.

В текущей версии файл хранится в базе данных в поле `fileData` как base64-строка. Вместе с ним сохраняются `originalFilename`, `mimeType`, `fileSize`, `title`, `content` и `summary`. Его ограничение состоит в том, что база данных начинает хранить не только метаданные, но и содержимое файлов. В дальнейшем для более крупной версии можно вынести файлы в S3/MinIO или другое объектное хранилище, оставив в базе только ссылки и метаданные.

Для поддерживаемых форматов backend пытается извлечь текст. Модуль `file-extraction.ts` обрабатывает текстовые файлы, JSON, XML, YAML, CSV, Markdown, HTML, CSS, JS, TS, PDF и DOCX. Для PDF используется `pdf-parse`, для DOCX - `mammoth`. Максимальный объем извлеченного текста ограничен 300000 символами. Если формат не поддерживается, файл все равно сохраняется, но summary получает статус `extractionStatus:unsupported`. Если извлечение завершилось ошибкой, статус фиксируется как `failed`.

Предпросмотр проектируется на стороне frontend. В разделе файлов пользователь видит список загруженных объектов, метаданные и доступные действия. Для некоторых типов может быть показан визуальный или текстовый предпросмотр, а для остальных файл остается доступным как объект с именем, размером и MIME-типом. При вопросе ассистенту по файлу frontend может передать идентификатор вложения, а backend повторно проверяет, что файл принадлежит текущему пользователю.

## 2.7. Проектирование пользовательского интерфейса

Пользовательский интерфейс проектируется как набор страниц внутри защищенного layout. После входа пользователь попадает в основную часть приложения. Главный экран связан с чатом. На нем пользователь вводит сообщение, прикрепляет файл, получает ответ ассистента и видит источники, если backend передал metadata с релевантными объектами. Чат также используется для suggested actions и для перехода к объектам, которые были найдены или созданы.

Страница заметок проектируется для работы с объектами типа `note`. Страница файлов работает с объектами типа `file`. Она должна поддерживать загрузку, отображение карточек, поиск, предпросмотр, скачивание, перенос в папку и удаление. Страница напоминаний работает с объектами типа `reminder`, отображает дату и статус, позволяет создать напоминание и отметить его выполненным. Страница папки `/folders/:id` показывает объекты, отфильтрованные по конкретной папке.

Страница настроек содержит профиль пользователя, действия выхода и статистику рабочего пространства. Статистика получается через endpoints `/stats/workspace`, `/stats/recent` и `/stats/reminders/upcoming`.

## 2.8. Проектирование подсистемы безопасности

Подсистема безопасности в текущей версии строится вокруг регистрации, входа пользователя, JWT-токена и проверки принадлежности данных пользователю. При проектировании учитываются типовые риски веб-приложений, описанные в материалах OWASP [16]. Для пользовательского интерфейса также важны понятность навигации и доступность основных действий на разных экранах; в качестве ориентира используются общие принципы WCAG [18]. При регистрации backend проверяет уникальность email, приводит email к нижнему регистру, хеширует пароль через `bcryptjs` и создает пользователя. Пароль в открытом виде в базе данных не хранится.

После успешной регистрации или входа backend возвращает JWT-токен. Токен формируется в `middlewares/auth.ts` и содержит `userId` и `email`. Срок действия токена - 30 дней. Frontend сохраняет токен в `localStorage` под ключом `mindvault_token` и использует его для запросов к защищенным API.

Защищенные endpoints используют middleware `requireAuth`. Он проверяет наличие заголовка `Authorization: Bearer ...`, валидирует токен и записывает данные пользователя в `req.auth`. Если токен отсутствует или недействителен, backend возвращает статус 401. На frontend защищенные маршруты также проверяют наличие токена и перенаправляют пользователя на `/auth`.

Разграничение данных выполняется на уровне запросов к базе данных. При получении, изменении и удалении объектов backend добавляет условие `userId = req.auth.userId`. Для папок также проверяется принадлежность папки текущему пользователю. При загрузке файла с `folderId` backend проверяет, что указанная папка принадлежит пользователю. Это защищает от доступа к чужим объектам через прямую подмену идентификатора в запросе.

В текущей версии есть ограничения, которые необходимо учитывать. Токен хранится в `localStorage`, refresh tokens не реализованы, ролей администратора нет, end-to-end encryption файлов отсутствует, production-контур с HTTPS/nginx не описан как реализованный. Эти решения допустимы для учебного MVP, но для промышленной эксплуатации потребовались бы дополнительные меры: более строгая политика хранения токенов, ротация ключей, аудит действий, ограничение размера файлов, проверка MIME-типов, rate limiting и полноценная настройка HTTPS.

## 3.4. Реализация базы данных

Для хранения данных используется PostgreSQL. Схема базы данных описана в пакете `lib/db` с помощью Drizzle ORM. Вместо ручного написания SQL-структуры в коде определены TypeScript-модули `users.ts`, `folders.ts`, `items.ts`, `conversations.ts` и `messages.ts`, которые затем используются backend-сервером.

В проекте определены следующие основные таблицы:

- `users` - учетные записи пользователей;
- `folders` - папки пользователя;
- `items` - универсальные объекты: заметки, файлы, напоминания и списки;
- `conversations` - диалоги пользователя с ассистентом;
- `messages` - сообщения внутри диалогов.

Таблица `users` содержит email, хеш пароля, имя пользователя, ссылку на аватар и даты создания/обновления. Email имеет ограничение уникальности, что предотвращает регистрацию нескольких пользователей с одним адресом.

Таблица `folders` связана с `users` через поле `userId`. При удалении пользователя его папки удаляются каскадно. Для папки хранятся название, цвет, иконка и признак `isSystem`. Системные папки создаются при регистрации пользователя и защищены от изменения и удаления.

Таблица `items` является центральной для предметной области приложения. Она содержит внешний ключ `userId`, необязательную ссылку `folderId`, перечисление `type`, заголовок, содержание, краткое описание, файловые поля, дату напоминания, статус и AI-метаданные. Для поля `type` используется перечисление `item_type` со значениями `note`, `file`, `reminder`, `list`. Для поля `status` используется перечисление `item_status` со значениями `active`, `archived`, `completed`.

Связь `items.folderId` настроена с поведением `set null`. Это означает, что при удалении папки сами объекты не удаляются, а остаются в рабочей области пользователя без привязки к удаленной папке. Такая логика реализована также на уровне маршрута удаления папки: перед удалением backend снимает привязку объектов к папке.

Таблицы `conversations` и `messages` используются для хранения истории чата. Диалог связан с пользователем, а сообщение связано с диалогом. При удалении диалога его сообщения удаляются каскадно. В сообщениях хранится роль (`user` или `assistant`), текст сообщения и поле `metadata` типа JSONB для дополнительных данных.

Подключение к базе данных выполняется в `lib/db/src/index.ts` через `drizzle-orm/node-postgres`. Для применения схемы при запуске в Docker используется команда `pnpm --filter @workspace/db run push`, указанная в `docker-compose.yml` для backend-контейнера.

## 3.5. Реализация ИИ-ассистента

ИИ-ассистент в MindVault реализован как часть backend-сервера. Внешний интерфейс для пользователя находится на главной странице приложения, а серверная логика расположена в маршрутах `artifacts/api-server/src/routes/gemini.ts`, модуле контекста `artifacts/api-server/src/lib/ai-context.ts`, модуле команд `artifacts/api-server/src/assistant/actions.ts` и клиенте AI-интеграции `lib/integrations-gemini-ai/src/client.ts`.

Несмотря на название маршрутов `gemini`, текущая интеграция использует OpenAI/OpenRouter-совместимый API формата Chat Completions. Провайдер выбирается по переменным окружения. Если задан `OPENROUTER_API_KEY`, используется OpenRouter; если задан `OPENAI_API_KEY`, используется OpenAI-совместимый endpoint. Для OpenAI по умолчанию указан `gpt-4.1-mini`, для OpenRouter - `openai/gpt-4o-mini`.

Перед отправкой сообщения внешней модели сервер формирует пользовательский контекст. В него включаются последние заметки, файлы, списки, напоминания, папки и релевантные объекты рабочей области. За счет этого ассистент отвечает не только на основе текущего сообщения, но и с учетом сохраненной информации. При этом в работе следует писать именно о контекстной выборке из сохраненных данных, а не о семантическом или векторном поиске, поскольку отдельная векторная база и embeddings в коде не реализованы.

Важная часть ассистента - слой команд. Он обрабатывает действия, которые должны изменить данные пользователя: создание заметки, списка, напоминания, поиск объектов, создание папки, переименование папки, удаление объектов с подтверждением, перенос объекта в папку. Для таких сценариев используется серверная логика `classifyAndExecuteAssistantAction`. Разбор команд заметок, списков и напоминаний выполняется детерминированно, через функции из `command-parser.ts`. Это подтверждается автоматическими тестами.

В текущей версии нельзя утверждать, что классификация пользовательского сообщения полностью выполняется большой языковой моделью. В маршруте `/api/gemini/classify` сначала вызывается серверная функция `classifyAndExecuteAssistantAction`, после чего результат сразу возвращается клиенту. Код AI-классификации ниже этой точки фактически не используется. Поэтому в ВКР корректно описывать реализованный механизм как серверную обработку команд ассистента с использованием правил, подтверждений и контекста, а не как полноценную LLM-классификацию всех пользовательских намерений.

Если AI-провайдер недоступен, не задан ключ API или возникает ошибка генерации, сервер формирует fallback-ответ с помощью функции `buildOfflineAssistantReply`. Такой ответ сообщает пользователю, что внешний AI сейчас недоступен, и при возможности использует часть локального контекста. Это важно для устойчивости приложения: сбой внешнего сервиса не приводит к падению чата и не блокирует все остальные функции MindVault.

## 3.6. Реализация работы с файлами

Работа с файлами реализована через общий механизм объектов `items` и отдельный endpoint `/api/items/upload`. На frontend пользователь выбирает файл на странице файлов или внутри папки, после чего файл передается на backend в виде base64-строки вместе с именем, MIME-типом, размером и необязательным идентификатором папки.

На сервере файл обрабатывается в маршруте `artifacts/api-server/src/routes/items.ts`. После проверки входных данных вызывается функция `extractTextFromUpload` из `artifacts/api-server/src/lib/file-extraction.ts`. Она определяет тип файла и пытается извлечь текст, если формат поддерживается. Для обычных текстовых файлов используется декодирование из base64 в UTF-8. Для PDF подключается `pdf-parse` [22], для DOCX - библиотека `mammoth` [21].

При успешном извлечении текст сохраняется в поле `content`, а в поле `summary` записывается служебный статус обработки. Если формат не поддерживается, файл все равно сохраняется как объект типа `file`, но текстовое содержимое не заполняется. Так файл и его метаданные сохраняются без ложного вывода о том, что система умеет анализировать любой бинарный формат.

Файл хранится в базе данных в поле `fileData` в формате base64. Для учебного проекта и локального запуска это упрощает развертывание, поскольку не требуется отдельное файловое хранилище. Для промышленной эксплуатации такой подход стоит заменить на объектное хранилище или файловый сервис, а в PostgreSQL оставлять только метаданные и ссылку на файл.

ИИ-ассистент может использовать текстовое содержимое файла при ответе, если текст был извлечен заранее или может быть извлечен лениво при формировании контекста. Если извлечь текст невозможно, ассистент должен опираться только на имя файла и метаданные. OCR для изображений и сканов в текущем коде не реализован.

## 3.7. Реализация заметок, папок, напоминаний и списков

Заметки, папки, файлы, напоминания и списки реализованы как связанные части одной рабочей области пользователя. Доступ к данным всегда ограничивается `userId`, который определяется из JWT-токена. За счет этого пользователь получает только свои папки, объекты, диалоги и сообщения.

Заметки. Заметка хранится как объект `items` с типом `note`. Для нее используются поля `title`, `content`, `folderId`, `status`, даты создания и обновления. Создание заметки возможно как через страницу заметок, так и через чат-команду, начинающуюся с ключевого слова. Старый режим автоматического сохранения произвольных фраз как заметок отключен через константу `LEGACY_AUTO_SAVE_ENABLED = false`, поэтому обычный вопрос пользователя не превращается в запись без явного намерения.

Папки. Папки хранятся в таблице `folders`. При регистрации пользователя backend создает набор системных папок. Пользовательские папки можно создавать, переименовывать, менять им цвет и иконку, а также удалять. Системные папки защищены: маршруты изменения и удаления проверяют поле `isSystem` и не позволяют выполнять эти операции над системными папками. При удалении обычной папки объекты не теряются, а остаются без папки.

Файлы. Файл хранится как объект `items` с типом `file`. Для него дополнительно сохраняются `originalFilename`, `mimeType`, `fileSize` и `fileData`. Файл может быть привязан к папке, отображаться на странице файлов или внутри конкретной папки. Пользователь может задавать ассистенту вопрос по файлу, если файл выбран в интерфейсе.

Напоминания. Напоминание хранится как объект `items` с типом `reminder`. Основное отличие от заметки - наличие поля `reminderAt`, в котором хранится дата и время напоминания. Статус `completed` используется для отметки выполненного напоминания. В текущей версии напоминания отображаются в приложении и участвуют в ближайших событиях, но push-уведомления и фоновые системные уведомления не реализованы.

Списки. Список хранится как объект `items` с типом `list`. Его пункты хранятся в поле `content` в виде JSON-структуры. На frontend список отображается как набор пунктов с чекбоксами. Изменение состояния пункта приводит к обновлению объекта через API. Для текущего масштаба приложения такой вариант проще отдельной таблицы пунктов списка.

Поиск. Поиск реализован через endpoint `/api/search`. Он использует текстовое сопоставление по заголовку, содержимому, краткому описанию и оригинальному имени файла. В коде используется `ilike`, поэтому поиск является обычным текстовым поиском по строкам. Семантический поиск, embeddings и векторное ранжирование в текущей версии отсутствуют.

## 3.8. Сборка и запуск проекта

Проект можно запускать локально через пакетный менеджер `pnpm` или через Docker Compose. В корневом `package.json` определены общие команды `build`, `typecheck` и `typecheck:libs`. В backend-приложении доступны команды `dev`, `build`, `start`, `typecheck`, `test` и `test:commands`. Во frontend-приложении доступны команды `dev`, `build`, `serve` и `typecheck`.

Для контейнерного запуска используется файл `docker-compose.yml`. Он описывает три сервиса:

- `db` - PostgreSQL 16 с базой `mindvault`;
- `api` - backend-сервер на порту `8080`;
- `web` - frontend-приложение на порту `18174`.

Сервис `api` зависит от готовности базы данных и перед запуском применяет схему командой `pnpm --filter @workspace/db run push`. Затем запускается backend в режиме разработки. Сервис `web` запускает Vite dev server и проксирует API-запросы на backend-сервис внутри Docker-сети.

## 3.10. Ограничения текущей версии и перспективы развития

Текущая версия MindVault закрывает основные сценарии учебного проекта: регистрацию и вход, работу с заметками, папками, файлами, напоминаниями, списками, чатом и контекстным ответом ассистента. При этом ряд функций находится за пределами реализованного кода. Их нельзя описывать как готовые возможности системы; корректнее вынести такие пункты в ограничения и перспективы развития.

Ограничения текущей версии и возможные направления развития представлены в табл. 13.

Таблица 13 - Ограничения текущей версии и перспективы развития

| Ограничение | Причина | Возможное развитие |

## Фрагмент 2. Middleware `requireAuth`

Файл: `artifacts/api-server/src/middlewares/auth.ts`.

Фрагмент подтверждает, что backend проверяет JWT-токен из заголовка `Authorization` и добавляет данные пользователя в `req.auth`.

```ts
export function generateToken(payload: AuthPayload): string {
  return jwt.sign(payload, JWT_SECRET, { expiresIn: "30d" });
}

export function verifyToken(token: string): AuthPayload | null {
  try {
    return jwt.verify(token, JWT_SECRET) as AuthPayload;
  } catch {
    return null;
  }
}

export function requireAuth(req: Request, res: Response, next: NextFunction): void {
  const authHeader = req.headers.authorization;

  if (!authHeader || !authHeader.startsWith("Bearer ")) {
    res.status(401).json({ error: "Authorization required" });
    return;
  }

  const token = authHeader.slice(7);
  const payload = verifyToken(token);

  if (!payload) {
    res.status(401).json({ error: "Invalid or expired token" });
    return;
  }

  req.auth = payload;
  next();
}
```

## Фрагмент 4. Извлечение текста из загруженного файла

```ts
export async function extractTextFromUpload(
  filename: string,
  mimeType: string,
  fileData: string,
): Promise<FileExtractionResult> {
  const buffer = Buffer.from(fileData, "base64");

  try {
    if (isPlainTextUpload(filename, mimeType)) {
      const text = validateTextLike(buffer.toString("utf8"));
      return text
        ? { text, status: "completed", summary: extractionSummary("completed", "plain text extracted") }
        : { text: null, status: "failed", summary: extractionSummary("failed", "plain text could not be decoded safely") };
    }

    if (isPdfUpload(filename, mimeType)) {
      const parsed = await loadPdfParser()(buffer);
      const text = cleanExtractedText(parsed.text);
      return text
        ? { text, status: "completed", summary: extractionSummary("completed", "pdf text extracted") }
        : { text: null, status: "failed", summary: extractionSummary("failed", "pdf parser returned no text") };
    }

    if (isDocxUpload(filename, mimeType)) {
      const parsed = await loadMammoth().extractRawText({ buffer });
      const text = cleanExtractedText(parsed.value);
      return text
        ? { text, status: "completed", summary: extractionSummary("completed", "docx text extracted") }
        : { text: null, status: "failed", summary: extractionSummary("failed", "docx parser returned no text") };
    }

    return { text: null, status: "unsupported", summary: extractionSummary("unsupported", "format is metadata-only") };
  } catch (err) {
    const message = err instanceof Error ? err.message.slice(0, 180) : "unknown extraction error";
    return { text: null, status: "failed", summary: extractionSummary("failed", message) };
  }
}
```

# ПРИЛОЖЕНИЕ 4. Инструкция по запуску приложения через Docker

## Предварительные требования

Для запуска приложения через Docker необходимо установить Docker и Docker Compose. Для локального режима разработки без контейнеров также потребуются Node.js и `pnpm`, однако при запуске через `docker compose` основные сервисы запускаются внутри контейнеров.

Перед запуском рекомендуется проверить наличие файла `.env` или переменных окружения. В текущем `docker-compose.yml` часть переменных имеет значения по умолчанию, но для полноценной работы AI-ассистента необходимо задать ключ одного из AI-провайдеров.

## Основные команды

1. Собрать и запустить контейнеры:

```bash
docker compose up -d --build
```

2. Проверить состояние контейнеров:

```bash
docker compose ps
```

3. Проверить логи backend:

```bash
docker compose logs api
```

4. Проверить логи frontend:

```bash
docker compose logs web
```

5. Открыть frontend-приложение:

```text
http://localhost:18174
```

6. Проверить доступность backend:

```text
http://localhost:8080/api/healthz
```

7. Остановить контейнеры:

```bash
docker compose down
```

## Сервисы Docker Compose

В файле `docker-compose.yml` описаны три сервиса:

- `db` - база данных PostgreSQL 16, доступная на порту `5432`;
- `api` - backend-сервер Express, доступный на порту `8080`;
- `web` - frontend-приложение Vite, доступное на порту `18174`.

Сервис `api` ожидает готовности базы данных, после чего применяет схему командой:

```bash
pnpm --filter @workspace/db run push
```

Затем backend запускается командой:

```bash
pnpm --filter @workspace/api-server run dev
```

Сервис `web` запускает frontend:

```bash
pnpm --filter @workspace/mindvault run dev
```

## Переменные окружения

В текущей конфигурации используются следующие переменные:

- `PORT` - порт backend-сервера внутри контейнера;
- `NODE_ENV` - режим работы приложения;
- `SESSION_SECRET` - секрет для подписи JWT;
- `DATABASE_URL` - строка подключения к PostgreSQL;
- `OPENAI_BASE_URL` - базовый URL OpenAI-совместимого API;
- `OPENAI_MODEL` - модель OpenAI по умолчанию;
- `OPENAI_API_KEY` - ключ OpenAI API;
- `OPENROUTER_BASE_URL` - базовый URL OpenRouter API;
- `OPENROUTER_MODEL` - модель OpenRouter по умолчанию;
- `OPENROUTER_API_KEY` - ключ OpenRouter API;
- `OPENROUTER_SITE_URL` - URL приложения для OpenRouter;
- `OPENROUTER_APP_NAME` - название приложения для OpenRouter;
- `ASSISTANT_SYSTEM_PROMPT` - необязательное системное сообщение ассистента;
- `BASE_PATH` - базовый путь frontend-приложения;
- `API_PROXY_TARGET` - адрес backend для проксирования API-запросов из Vite.

Если `OPENAI_API_KEY` и `OPENROUTER_API_KEY` не заданы, основные функции приложения продолжают работать: регистрация, вход, заметки, файлы, папки, напоминания, списки и поиск не требуют внешнего AI API. Чат в этом случае не сможет получить ответ от внешней модели и должен вернуть fallback-сообщение.

Текущий Docker Compose используется для локального и демонстрационного запуска. Он не описывает production-деплой с HTTPS, nginx, доменным именем и отдельной инфраструктурой мониторинга.

Файл `docker-compose.yml` описывает запуск PostgreSQL, backend и frontend. Файл `TESTING.md` содержит ручные сценарии проверки приложения, которые следует пройти перед финальной сдачей ВКР.
