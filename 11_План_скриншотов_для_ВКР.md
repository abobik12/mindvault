# План скриншотов для ВКР MindVault

Документ описывает подготовку реальных скриншотов для рисунков 7-15. DOCX на этом этапе не создается; скриншоты вставляются в Word вручную после запуска приложения и заполнения демо-данных.

## Предварительная подготовка

1. Запустить проект через Docker Compose:

```bash
docker compose up -d --build
```

2. Проверить состояние контейнеров:

```bash
docker compose ps
```

3. Открыть приложение:

```text
http://localhost:18174
```

4. Зарегистрировать тестового пользователя и внести демо-данные из файла `12_Демо_данные_для_скриншотов.md`.

5. Перед съемкой закрыть лишние вкладки браузера, убрать панель закладок и не показывать личные данные, ключи API, `.env`, логи с секретами и реальные файлы пользователя.

## Рис. 7. Главная страница приложения MindVault

- Экран: главная страница с чат-интерфейсом и боковым меню.
- URL: `http://localhost:18174/`
- Ширина: обычная desktop-ширина, примерно 1366x768 или 1440x900.
- Данные на экране: боковое меню MindVault, разделы "Чат", "Заметки", "Списки", "Файлы", "Напоминания", созданные папки, область чата, поле ввода сообщения.
- Что должно попасть в кадр: общий вид приложения после входа, навигация слева, рабочая область чата, несколько последних сообщений или стартовый экран чата.
- Подпись в Word: `Рис. 7. Главная страница приложения MindVault`
- Не показывать: пустую неавторизованную страницу, браузерные расширения, личную почту, консоль разработчика, ошибки API.

## Рис. 8. Чат с ИИ-ассистентом

- Экран: главная страница с активным диалогом.
- URL: `http://localhost:18174/`
- Ширина: обычная desktop-ширина.
- Данные на экране: 3-5 сообщений в чате, пример вопроса по сохраненным данным и ответ ассистента со ссылкой на релевантные заметки или файлы, если они отображаются интерфейсом.
- Что должно попасть в кадр: сообщение пользователя, ответ ассистента, поле ввода, кнопки создания/подсказок, если они появились после неоднозначного ввода.
- Подпись в Word: `Рис. 8. Чат с ИИ-ассистентом`
- Не показывать: сообщения с ошибками внешнего AI API, ключи доступа, слишком длинный ответ, который превращает кадр в нечитаемую ленту.

## Рис. 9. Страница заметок

- Экран: раздел заметок.

- Что должно попасть в кадр: заголовок "Заметки", карточки заметок, краткие фрагменты содержания, боковое меню.
- Подпись в Word: `Рис. 9. Страница заметок`
- Не показывать: модальное окно удаления, пустой список, заметки с черновым текстом "test", "123", личные данные.

## Рис. 10. Страница файлов

- Экран: раздел файлов.
- URL: `http://localhost:18174/files`
- Ширина: обычная desktop-ширина.
- Данные на экране: 2-3 файла разных типов, например TXT/MD, PDF и DOCX, строка поиска, выбор папки для загрузки, кнопка загрузки.
- Что должно попасть в кадр: заголовок "Файлы", карточки файлов с именами, типами и размерами, элементы управления загрузкой и поиска.
- Подпись в Word: `Рис. 10. Страница файлов`
- Не показывать: реальные документы пользователя, файлы с персональными данными, неподдерживаемые бинарные файлы как основной пример, ошибки извлечения текста.

## Рис. 11. Страница напоминаний

- Экран: раздел напоминаний.
- URL: `http://localhost:18174/reminders`
- Ширина: обычная desktop-ширина.
- Данные на экране: 2-3 напоминания с будущими датами и понятными названиями.
- Что должно попасть в кадр: заголовок "Напоминания", блок ближайших напоминаний, общий список напоминаний, кнопка создания.
- Подпись в Word: `Рис. 11. Страница напоминаний`
- Не показывать: просроченные или случайные напоминания, личные встречи, медицинские или финансовые данные.

## Рис. 12. Страница списков

- Экран: раздел списков.
- URL: `http://localhost:18174/lists`
- Ширина: обычная desktop-ширина.
- Данные на экране: 2 списка с несколькими пунктами, один пункт можно отметить выполненным.
- Что должно попасть в кадр: заголовок "Списки", карточки списков, чекбоксы пунктов, кнопка создания списка, поиск.
- Подпись в Word: `Рис. 12. Страница списков`
- Не показывать: список покупок как единственный пример, слишком длинные пункты, личные бытовые данные.

## Рис. 13. Отображение приложения на мобильном устройстве

- Экран: мобильная версия одного из основных разделов, предпочтительно чат или заметки.
- URL: `http://localhost:18174/`
- Ширина: мобильная, например 390x844 или 375x812 в режиме Device Toolbar браузера.
- Данные на экране: мобильная верхняя панель MindVault, доступ к навигации, область чата или карточки заметок без горизонтального скролла.
- Что должно попасть в кадр: адаптивная компоновка, отсутствие наложения текста, доступное поле ввода или список карточек.
- Подпись в Word: `Рис. 13. Отображение приложения на мобильном устройстве`
- Не показывать: desktop-версию, просто сжатую до маленького окна; открытые системные панели браузера; элементы, перекрывающие поле ввода.

## Рис. 14. Результат запуска контейнеров Docker

- Экран: терминал с результатом запуска и проверки контейнеров.
- Команды:

```bash

docker compose ps
```

- Ширина: обычная desktop-ширина терминала.
- Данные на экране: контейнеры `web`, `api`, `db`, проброшенные порты `18174`, `8080`, `5432`, состояние `running` или `Up`.
- Что должно попасть в кадр: выполненная команда `docker compose ps` и таблица со статусом сервисов.
- Подпись в Word: `Рис. 14. Результат запуска контейнеров Docker`
- Не показывать: `.env`, API-ключи, длинные логи сборки, ошибки старта контейнеров.

## Рис. 15. Результат выполнения тестов

- Экран: терминал с результатами автоматических проверок.
- Актуальные команды по текущему `package.json`:

```bash
corepack pnpm --filter @workspace/api-server test
corepack pnpm --filter @workspace/api-server run typecheck
corepack pnpm --filter @workspace/mindvault run typecheck
```

- Дополнительно можно показать общую проверку workspace:

```bash
corepack pnpm run typecheck
```

- Ширина: обычная desktop-ширина терминала.
- Данные на экране: успешное выполнение теста `command-parser.test.ts` и успешные TypeScript-проверки backend/frontend.
- Что должно попасть в кадр: команды и финальные строки без ошибок.
- Подпись в Word: `Рис. 15. Результат выполнения тестов`
- Не показывать: устаревшие команды из `TESTING.md` с filter-именами `@mindvault/api-server` и `mindvault`, если они не соответствуют текущим пакетам; фрагменты лога с секретами окружения.

## Команды для подготовки и проверки

Запуск проекта:

```bash
docker compose up -d --build
```

Проверка контейнеров:

```bash
docker compose ps
docker compose logs api
docker compose logs web
```

Backend tests:

```bash
corepack pnpm --filter @workspace/api-server test
```

Backend typecheck:

```bash
corepack pnpm --filter @workspace/api-server run typecheck
```

Frontend typecheck:

```bash
corepack pnpm --filter @workspace/mindvault run typecheck
```

Общая TypeScript-проверка workspace:

```bash
corepack pnpm run typecheck
```

## Порядок съемки

1. Запустить Docker Compose и сделать скриншот `docker compose ps` для рисунка 14.
2. Выполнить backend tests и typecheck-команды, сделать скриншот терминала для рисунка 15.
3. Открыть `http://localhost:18174`, зарегистрироваться или войти под тестовым пользователем.
4. Создать папки, заметки, списки, напоминания, файлы и сообщения по файлу `12_Демо_данные_для_скриншотов.md`.
5. Сделать desktop-скриншоты рисунков 7-12.
6. Включить мобильную ширину 390x844 или 375x812 и сделать скриншот рисунка 13.
