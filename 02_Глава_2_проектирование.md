# ГЛАВА 2. ПРОЕКТИРОВАНИЕ ВЕБ-ПРИЛОЖЕНИЯ MINDVAULT

## 2.1. Общая концепция приложения

MindVault проектируется как кроссплатформенное веб-приложение "личный секретарь" для хранения и повторного использования персональной цифровой информации. В рамках работы под кроссплатформенностью понимается доступ через браузер на настольных и мобильных устройствах. Отдельное нативное мобильное приложение в текущую версию не входит.

Основная идея приложения состоит в объединении нескольких привычных типов данных в одном рабочем пространстве. Пользователь может хранить заметки, файлы, папки, напоминания и списки, а также взаимодействовать с ИИ-ассистентом через чат. При этом чат не заменяет базу данных и не является единственным интерфейсом. Он выступает как дополнительный способ ввода команд, поиска информации и получения ответов с учетом сохраненного пользовательского контекста.

В приложении предусмотрены два способа работы. Первый способ - классический интерфейс с отдельными разделами: "Чат", "Заметки", "Списки", "Файлы", "Напоминания", папки и настройки. Пользователь открывает нужный раздел и выполняет действие через форму, карточку или кнопку. Второй способ - чатовый интерфейс, где пользователь может задать вопрос ассистенту или ввести явную команду на создание объекта. Такой подход удобен для быстрого добавления информации, но при проектировании важно отделять обычный вопрос от действия, изменяющего данные.

MindVault обрабатывает следующие основные типы данных:

- учетные записи пользователей;
- папки пользователя;
- элементы хранилища с типами `note`, `file`, `reminder`, `list`;
- диалоги с ассистентом;
- сообщения в диалогах.

С точки зрения пользователя приложение решает несколько задач. Во-первых, оно дает единое место для хранения коротких текстов, файлов, списков и напоминаний. Во-вторых, оно позволяет разделять данные по папкам и искать сохраненные объекты. В-третьих, оно связывает ИИ-ассистента с пользовательским контекстом: при ответе ассистент может учитывать сохраненные заметки, файлы, папки, списки, напоминания и релевантные сообщения.

Границы текущей версии определены кодом проекта. В проектировании не рассматриваются как обязательные функции push-уведомления, OCR, semantic/vector search, HTTPS/nginx/production-деплой, полноценное мобильное приложение, совместная работа, роли администратора, оплата, подписки и интеграции с Telegram или Google Calendar. Эти направления могут быть описаны только как дальнейшее развитие.

## 2.3. Архитектура системы

Архитектура MindVault разделена на четыре основные части: frontend, backend, базу данных и внешний AI API. Такое разделение подходит для MVP, потому что каждая часть имеет понятную ответственность и может проверяться отдельно.

Frontend-часть расположена в `artifacts/mindvault`. Она реализована на React, TypeScript и Vite [2]. Навигация построена через Wouter. В файле `App.tsx` определены маршруты `/auth`, `/`, `/notes`, `/lists`, `/files`, `/reminders`, `/folders/:id` и `/settings`. Защищенные маршруты проверяют наличие токена `mindvault_token` в `localStorage`. Основной layout содержит боковую навигацию, список папок, переходы к разделам и выход из аккаунта.

Backend-часть расположена в `artifacts/api-server`. Она реализована на Express 5 и TypeScript [3]. В `app.ts` подключены логирование, CORS, JSON parser с лимитом 25 МБ и общий роутер `/api`. Роуты разделены по файлам: `auth.ts`, `folders.ts`, `items.ts`, `gemini.ts`, `health.ts`. Сервер получает порт из переменной окружения `PORT`.

База данных описана в `lib/db/src/schema`. Для работы с PostgreSQL используется Drizzle ORM [4], [6]. В проекте есть таблицы пользователей, папок, элементов хранилища, диалогов и сообщений. Сущность `items` используется как универсальная таблица для заметок, файлов, напоминаний и списков. Такой подход упрощает общую фильтрацию и поиск по объектам, но требует аккуратной работы с полями, которые относятся только к отдельным типам.

Внешний AI API подключается через модуль `lib/integrations-gemini-ai/src/client.ts`. Несмотря на название интеграционного пакета, фактический клиент использует OpenAI/OpenRouter-compatible endpoint `/chat/completions`. Провайдер выбирается по переменным окружения `OPENROUTER_API_KEY` и `OPENAI_API_KEY`. По умолчанию для OpenAI указан `gpt-4.1-mini`, для OpenRouter - `openai/gpt-4o-mini` [7].

Таблица 9 - Описание сущностей базы данных

| Сущность | Назначение | Основные поля | Связи |
|---|---|---|---|
| `users` | Хранение учетных записей пользователей. | `id`, `email`, `passwordHash`, `fullName`, `avatarUrl`, `createdAt`, `updatedAt`. | Один пользователь связан с папками, элементами и диалогами. |
| `folders` | Хранение пользовательских и системных папок. | `id`, `userId`, `name`, `color`, `icon`, `isSystem`, `createdAt`, `updatedAt`. | `userId` ссылается на `users.id`; папка может быть связана с элементами через `items.folderId`. |
| `items` | Универсальное хранение заметок, файлов, напоминаний и списков. | `id`, `userId`, `folderId`, `type`, `title`, `content`, `summary`, `originalFilename`, `mimeType`, `fileSize`, `fileData`, `reminderAt`, `status`, `aiCategory`, `aiTags`, `aiConfidence`, `createdAt`, `updatedAt`. | `userId` ссылается на `users.id`; `folderId` ссылается на `folders.id` и при удалении папки устанавливается в `null`. |
| `conversations` | Хранение диалогов пользователя с ассистентом. | `id`, `userId`, `title`, `createdAt`, `updatedAt`. | `userId` ссылается на `users.id`; один диалог содержит много сообщений. |
| `messages` | Хранение сообщений в диалогах. | `id`, `conversationId`, `role`, `content`, `metadata`, `createdAt`. | `conversationId` ссылается на `conversations.id`; при удалении диалога сообщения удаляются каскадно. |

В таблице `items` поле `type` ограничено значениями `note`, `file`, `reminder`, `list`. Поле `status` принимает значения `active`, `archived`, `completed`. Такой вариант проектирования позволяет использовать единые маршруты `/items` для разных типов объектов. Например, заметка и список различаются значением `type` и содержанием `content`, файл дополнительно использует поля `originalFilename`, `mimeType`, `fileSize` и `fileData`, а напоминание использует поле `reminderAt`.

Связь `folders` и `items` спроектирована мягко: при удалении папки связанные элементы не удаляются, а их `folderId` становится `null`. Это важно для пользовательского сценария, потому что удаление папки не должно автоматически уничтожать заметки и файлы. Для системных папок предусмотрено поле `isSystem`, а backend запрещает их изменение и удаление.

Для таблицы `messages` предусмотрено поле `metadata`. Оно используется для хранения дополнительной информации о сообщении: результата action-логики, sources, вложений и других данных, которые не должны быть отдельными таблицами в текущей версии. Такой подход удобен для MVP, но в дальнейшем отдельные типы metadata можно вынести в более строгие сущности.

## 2.5. Проектирование ИИ-ассистента

ИИ-ассистент проектируется как часть серверной логики, связанная с чатами, сообщениями и сохраненными данными пользователя. Frontend отправляет сообщение в backend, backend сохраняет пользовательское сообщение, определяет возможные действия, формирует контекст и обращается к внешнему AI API. Ответ ассистента также сохраняется в таблице `messages`.

[ВСТАВИТЬ РИС. 4. Схема обработки пользовательского сообщения]

Рис. 4. Схема обработки пользовательского сообщения

На рис. 4 следует показать путь сообщения от формы чата во frontend до endpoint `/api/gemini/conversations/:id/messages`, сохранение сообщения в базе данных, вызов action-логики, формирование контекста и получение ответа от AI-провайдера.

В проекте предусмотрены два слоя обработки сообщения. Первый слой - серверная action-логика в `assistant/actions.ts` и `assistant/command-parser.ts`. Она обрабатывает явные команды, поиск по пользовательским данным, создание заметок, списков, напоминаний, создание и изменение папок, перенос объектов, удаление объектов с подтверждением и очистку истории чата. Часть действий определяется до обращения к AI. Это сделано для того, чтобы операции с базой данных выполнялись предсказуемо и могли быть подтверждены только после успешного результата.

Второй слой - обращение к AI API. Он используется для обычного ответа в чате и для работы с сохраненным пользовательским контекстом. Контекст формируется в `lib/ai-context.ts`. Backend собирает количество объектов, папки, последние заметки, файлы, списки, ближайшие напоминания и релевантные источники. Поиск релевантных источников основан на текстовом сопоставлении по заголовкам, содержимому, summary, имени файла, папке, тегам и типу объекта. Это не semantic/vector search, а обычный контекстный подбор по сохраненным данным.

На рис. 5 должны быть показаны таблицы `items`, `folders`, `messages`, модуль `buildAssistantContext`, prompt для AI API и ответ ассистента. Важно отразить, что контекст принадлежит только текущему авторизованному пользователю и отбирается backend-логикой.

Для явных команд используется parser. Например, сообщение, начинающееся с ключевых слов для заметки, напоминания или списка, может быть обработано как команда создания объекта. В таком случае пользователь выбирает, сохранить сообщение как заметку, создать напоминание, создать список или оставить его обычным сообщением.

При ошибке AI-провайдера предусмотрен fallback. В `gemini.ts` есть функция формирования офлайн-ответа. Если внешний API недоступен или возвращает ошибку лимита, backend сохраняет резервный ответ ассистента и сообщает пользователю причину. Если возможно, в ответ включается краткая сводка по найденному пользовательскому контексту. Это позволяет не блокировать работу всего приложения из-за ошибки внешнего сервиса.

## 2.6. Проектирование работы с файлами

Работа с файлами проектируется как часть общей модели `items`. Файл является объектом типа `file`. Frontend передает файл в backend через endpoint `/api/items/upload`. В запрос входят имя файла, MIME-тип, размер, base64-представление содержимого и при необходимости `folderId`.

[ВСТАВИТЬ РИС. 6. Схема загрузки и обработки файла]

Рис. 6. Схема загрузки и обработки файла

На рис. 6 нужно показать выбор файла во frontend, преобразование в base64, отправку на `/api/items/upload`, извлечение текста для поддерживаемых форматов, сохранение метаданных и содержимого в таблице `items`, а также дальнейшее использование текста файла в поиске и контексте ассистента.

В текущей версии файл хранится в базе данных в поле `fileData` как base64-строка. Вместе с ним сохраняются `originalFilename`, `mimeType`, `fileSize`, `title`, `content` и `summary`. Его ограничение состоит в том, что база данных начинает хранить не только метаданные, но и содержимое файлов. В дальнейшем для более крупной версии можно вынести файлы в S3/MinIO или другое объектное хранилище, оставив в базе только ссылки и метаданные.

Для поддерживаемых форматов backend пытается извлечь текст. Модуль `file-extraction.ts` обрабатывает текстовые файлы, JSON, XML, YAML, CSV, Markdown, HTML, CSS, JS, TS, PDF и DOCX. Для PDF используется `pdf-parse`, для DOCX - `mammoth`. Максимальный объем извлеченного текста ограничен 300000 символами. Если формат не поддерживается, файл все равно сохраняется, но summary получает статус `extractionStatus:unsupported`. Если извлечение завершилось ошибкой, статус фиксируется как `failed`.

Предпросмотр проектируется на стороне frontend. В разделе файлов пользователь видит список загруженных объектов, метаданные и доступные действия. Для некоторых типов может быть показан визуальный или текстовый предпросмотр, а для остальных файл остается доступным как объект с именем, размером и MIME-типом. При вопросе ассистенту по файлу frontend может передать идентификатор вложения, а backend повторно проверяет, что файл принадлежит текущему пользователю.

## 2.7. Проектирование пользовательского интерфейса

Пользовательский интерфейс проектируется как набор страниц внутри защищенного layout. После входа пользователь попадает в основную часть приложения. Боковая навигация содержит разделы "Чат", "Заметки", "Списки", "Файлы", "Напоминания", список папок и переход к настройкам. Такая структура соответствует фактическим маршрутам frontend-приложения.

Главный экран связан с чатом. На нем пользователь вводит сообщение, прикрепляет файл, получает ответ ассистента и видит источники, если backend передал metadata с релевантными объектами. Чат также используется для suggested actions и для перехода к объектам, которые были найдены или созданы.

Страница заметок проектируется для работы с объектами типа `note`. Пользователь видит список заметок, может создать новую заметку, отредактировать заголовок и содержание, выбрать папку и удалить объект. Страница списков работает с объектами типа `list`; пункты списка хранятся в `content` как JSON-структура, поэтому интерфейс должен уметь отображать пункты и изменять их состояние.

Страница файлов работает с объектами типа `file`. Она должна поддерживать загрузку, отображение карточек, поиск, предпросмотр, скачивание, перенос в папку и удаление. Страница напоминаний работает с объектами типа `reminder`, отображает дату и статус, позволяет создать напоминание и отметить его выполненным. Страница папки `/folders/:id` показывает объекты, отфильтрованные по конкретной папке.

Страница настроек содержит профиль пользователя, действия выхода и статистику рабочего пространства. Статистика получается через endpoints `/stats/workspace`, `/stats/recent` и `/stats/reminders/upcoming`.

В интерфейсе предусмотрены пустые состояния: если в разделе нет объектов, пользователь должен видеть понятное сообщение и действие для создания первого объекта. Для демонстрации и будущих скриншотов интерфейс должен оставаться светлым и читаемым: это упростит перенос изображений в ВКР и позволит лучше показать структуру приложения.

## 2.8. Проектирование подсистемы безопасности

Подсистема безопасности в текущей версии строится вокруг регистрации, входа пользователя, JWT-токена и проверки принадлежности данных пользователю. При регистрации backend проверяет уникальность email, приводит email к нижнему регистру, хеширует пароль через `bcryptjs` и создает пользователя. Пароль в открытом виде в базе данных не хранится.

После успешной регистрации или входа backend возвращает JWT-токен. Токен формируется в `middlewares/auth.ts` и содержит `userId` и `email`. Срок действия токена - 30 дней. Frontend сохраняет токен в `localStorage` под ключом `mindvault_token` и использует его для запросов к защищенным API.

Защищенные endpoints используют middleware `requireAuth`. Он проверяет наличие заголовка `Authorization: Bearer ...`, валидирует токен и записывает данные пользователя в `req.auth`. Если токен отсутствует или недействителен, backend возвращает статус 401. На frontend защищенные маршруты также проверяют наличие токена и перенаправляют пользователя на `/auth`.

Разграничение данных выполняется на уровне запросов к базе данных. При получении, изменении и удалении объектов backend добавляет условие `userId = req.auth.userId`. Для папок также проверяется принадлежность папки текущему пользователю. При загрузке файла с `folderId` backend проверяет, что указанная папка принадлежит пользователю. Это защищает от доступа к чужим объектам через прямую подмену идентификатора в запросе.

В текущей версии есть ограничения, которые необходимо учитывать. Токен хранится в `localStorage`, refresh tokens не реализованы, ролей администратора нет, end-to-end encryption файлов отсутствует, production-контур с HTTPS/nginx не описан как реализованный. Эти решения допустимы для учебного MVP, но для промышленной эксплуатации потребовались бы дополнительные меры: более строгая политика хранения токенов, ротация ключей, аудит действий, ограничение размера файлов, проверка MIME-типов, rate limiting и полноценная настройка HTTPS.
