# ГЛАВА 3. РЕАЛИЗАЦИЯ И ТЕСТИРОВАНИЕ ВЕБ-ПРИЛОЖЕНИЯ MINDVAULT

## 3.1. Общая структура проекта

Практическая реализация приложения MindVault выполнена в виде монорепозитория на TypeScript. В одном рабочем пространстве размещены клиентская часть, серверная часть, описание базы данных, сгенерированные API-клиенты и вспомогательные библиотеки. Такой вариант структуры удобен для выпускной работы, поскольку позволяет рассматривать систему как единый программный продукт, но при этом разделять зоны ответственности между frontend, backend и слоем данных.

Корневая директория проекта содержит конфигурацию рабочего пространства `pnpm`, общий файл `package.json`, Docker-конфигурацию, файл `docker-compose.yml`, а также документ `TESTING.md` с чек-листом ручной проверки. Основные приложения находятся в каталоге `artifacts`: frontend-приложение расположено в `artifacts/mindvault`, backend-сервер - в `artifacts/api-server`. Общие библиотеки вынесены в каталог `lib`.

Структура проекта подтверждает, что MindVault реализован не как набор несвязанных прототипов, а как веб-приложение с отдельной клиентской частью, серверным API, схемой базы данных и интеграцией с внешним AI-провайдером. Основные компоненты проекта представлены в табл. 10.

Такой подход уменьшает количество однотипных таблиц и упрощает реализацию поиска, статистики и контекстной выборки для ассистента.

## 3.2. Реализация frontend-части

Клиентская часть MindVault реализована как одностраничное приложение на React. Точкой входа в маршрутизацию является файл `artifacts/mindvault/src/App.tsx`. В нем определены основные маршруты приложения: `/auth`, `/`, `/notes`, `/lists`, `/files`, `/reminders`, `/folders/:id` и `/settings`. Для защищенных страниц используется компонент `ProtectedRoute`, который проверяет наличие JWT-токена в `localStorage` по ключу `mindvault_token`. Если токен отсутствует, пользователь перенаправляется на страницу авторизации.

Страница авторизации расположена в `artifacts/mindvault/src/pages/auth.tsx`. Она использует хуки `useLogin` и `useRegister`, полученные из сгенерированного API-клиента. После успешного входа или регистрации токен сохраняется в `localStorage`, а данные текущего пользователя обновляются через TanStack Query. Такой механизм не является расширенной системой управления сессиями, но он обеспечивает базовую авторизацию для веб-приложения.

Основная компоновка интерфейса реализована в компоненте `artifacts/mindvault/src/components/layout.tsx`. В нем размещены боковое меню, мобильная навигация, ссылки на разделы приложения, список пользовательских папок, кнопки создания и редактирования папок, а также действие выхода из учетной записи. Навигация включает разделы чата, заметок, списков, файлов, напоминаний и настроек.

## 3.3. Реализация backend-части и API

Серверная часть приложения реализована на Express. Основной файл приложения - `artifacts/api-server/src/app.ts`. В нем подключаются логирование через `pino-http`, CORS, обработка JSON и URL-encoded тела запроса с лимитом `25mb`, после чего все маршруты монтируются по префиксу `/api`. Запуск сервера выполняется из `artifacts/api-server/src/index.ts`, где приложение начинает слушать порт, заданный переменной окружения `PORT`.

Маршруты backend разделены по предметным областям: авторизация, папки, универсальные объекты `items`, статистика, поиск и чат. Большинство маршрутов защищено middleware `requireAuth`, который проверяет JWT-токен из заголовка `Authorization: Bearer ...` и добавляет в запрос данные пользователя.

## 3.4. Реализация базы данных

Для хранения данных используется PostgreSQL. Схема базы данных описана в пакете `lib/db` с помощью Drizzle ORM. Вместо ручного написания SQL-структуры в коде определены TypeScript-модули `users.ts`, `folders.ts`, `items.ts`, `conversations.ts` и `messages.ts`, которые затем используются backend-сервером.

В проекте определены следующие основные таблицы:

- `users` - учетные записи пользователей;
- `folders` - папки пользователя;
- `items` - универсальные объекты: заметки, файлы, напоминания и списки;
- `conversations` - диалоги пользователя с ассистентом;
- `messages` - сообщения внутри диалогов.

Таблица `users` содержит email, хеш пароля, имя пользователя, ссылку на аватар и даты создания/обновления. Email имеет ограничение уникальности, что предотвращает регистрацию нескольких пользователей с одним адресом.

Таблица `folders` связана с `users` через поле `userId`. При удалении пользователя его папки удаляются каскадно. Для папки хранятся название, цвет, иконка и признак `isSystem`. Системные папки создаются при регистрации пользователя и защищены от изменения и удаления.

Таблица `items` является центральной для предметной области приложения. Она содержит внешний ключ `userId`, необязательную ссылку `folderId`, перечисление `type`, заголовок, содержание, краткое описание, файловые поля, дату напоминания, статус и AI-метаданные. Для поля `type` используется перечисление `item_type` со значениями `note`, `file`, `reminder`, `list`. Для поля `status` используется перечисление `item_status` со значениями `active`, `archived`, `completed`.

Связь `items.folderId` настроена с поведением `set null`. Это означает, что при удалении папки сами объекты не удаляются, а остаются в рабочей области пользователя без привязки к удаленной папке. Такая логика реализована также на уровне маршрута удаления папки: перед удалением backend снимает привязку объектов к папке.

Таблицы `conversations` и `messages` используются для хранения истории чата. Диалог связан с пользователем, а сообщение связано с диалогом. При удалении диалога его сообщения удаляются каскадно. В сообщениях хранится роль (`user` или `assistant`), текст сообщения и поле `metadata` типа JSONB для дополнительных данных.

Подключение к базе данных выполняется в `lib/db/src/index.ts` через `drizzle-orm/node-postgres`. Для применения схемы при запуске в Docker используется команда `pnpm --filter @workspace/db run push`, указанная в `docker-compose.yml` для backend-контейнера.

## 3.5. Реализация ИИ-ассистента

ИИ-ассистент в MindVault реализован как часть backend-сервера. Внешний интерфейс для пользователя находится на главной странице приложения, а серверная логика расположена в маршрутах `artifacts/api-server/src/routes/gemini.ts`, модуле контекста `artifacts/api-server/src/lib/ai-context.ts`, модуле команд `artifacts/api-server/src/assistant/actions.ts` и клиенте AI-интеграции `lib/integrations-gemini-ai/src/client.ts`.

Несмотря на название маршрутов `gemini`, текущая интеграция использует OpenAI/OpenRouter-совместимый API формата Chat Completions. Провайдер выбирается по переменным окружения. Если задан `OPENROUTER_API_KEY`, используется OpenRouter; если задан `OPENAI_API_KEY`, используется OpenAI-совместимый endpoint. Для OpenAI по умолчанию указан `gpt-4.1-mini`, для OpenRouter - `openai/gpt-4o-mini`.

Перед отправкой сообщения внешней модели сервер формирует пользовательский контекст. В него включаются последние заметки, файлы, списки, напоминания, папки и релевантные объекты рабочей области. Это позволяет ассистенту отвечать с учетом сохраненной информации, а не только текущего сообщения. При этом в работе следует писать именно о контекстной выборке из сохраненных данных, а не о семантическом или векторном поиске, поскольку отдельная векторная база и embeddings в коде не реализованы.

Важной частью ассистента является слой команд. Он обрабатывает действия, которые должны изменить данные пользователя: создание заметки, списка, напоминания, поиск объектов, создание папки, переименование папки, удаление объектов с подтверждением, перенос объекта в папку. Для таких сценариев используется серверная логика `classifyAndExecuteAssistantAction`. Разбор команд заметок, списков и напоминаний выполняется детерминированно, через функции из `command-parser.ts`. Это подтверждается автоматическими тестами.

В текущей версии нельзя утверждать, что классификация пользовательского сообщения полностью выполняется большой языковой моделью. В маршруте `/api/gemini/classify` сначала вызывается серверная функция `classifyAndExecuteAssistantAction`, после чего результат сразу возвращается клиенту. Код AI-классификации ниже этой точки фактически не используется. Поэтому в ВКР корректно описывать реализованный механизм как серверную обработку команд ассистента с использованием правил, подтверждений и контекста, а не как полноценную LLM-классификацию всех пользовательских намерений.

Если AI-провайдер недоступен, не задан ключ API или возникает ошибка генерации, сервер формирует fallback-ответ с помощью функции `buildOfflineAssistantReply`. Такой ответ сообщает пользователю, что внешний AI сейчас недоступен, и при возможности использует часть локального контекста. Это важно для устойчивости приложения: сбой внешнего сервиса не приводит к падению чата и не блокирует все остальные функции MindVault.

## 3.6. Реализация работы с файлами

Работа с файлами реализована через общий механизм объектов `items` и отдельный endpoint `/api/items/upload`. На frontend пользователь выбирает файл на странице файлов или внутри папки, после чего файл передается на backend в виде base64-строки вместе с именем, MIME-типом, размером и необязательным идентификатором папки.

На сервере файл обрабатывается в маршруте `artifacts/api-server/src/routes/items.ts`. После проверки входных данных вызывается функция `extractTextFromUpload` из `artifacts/api-server/src/lib/file-extraction.ts`. Она определяет тип файла и пытается извлечь текст, если формат поддерживается. Для обычных текстовых файлов используется декодирование из base64 в UTF-8. Для PDF подключается `pdf-parse`, для DOCX - библиотека `mammoth`.

При успешном извлечении текст сохраняется в поле `content`, а в поле `summary` записывается служебный статус обработки. Если формат не поддерживается, файл все равно сохраняется как объект типа `file`, но текстовое содержимое не заполняется. Такой вариант позволяет хранить файл и его метаданные, не создавая ложного впечатления, что система умеет анализировать любой бинарный формат.

Файл хранится в базе данных в поле `fileData` в формате base64. Для учебного проекта и локального запуска это упрощает развертывание, поскольку не требуется отдельное файловое хранилище. Для промышленной эксплуатации такой подход стоит заменить на объектное хранилище или файловый сервис, а в PostgreSQL оставлять только метаданные и ссылку на файл.

ИИ-ассистент может использовать текстовое содержимое файла при ответе, если текст был извлечен заранее или может быть извлечен лениво при формировании контекста. Если извлечь текст невозможно, ассистент должен опираться только на имя файла и метаданные. OCR для изображений и сканов в текущем коде не реализован.

## 3.7. Реализация заметок, папок, напоминаний и списков

Заметки, папки, файлы, напоминания и списки реализованы как связанные части одной рабочей области пользователя. Доступ к данным всегда ограничивается `userId`, который определяется из JWT-токена. За счет этого пользователь получает только свои папки, объекты, диалоги и сообщения.

Заметки. Заметка является объектом `items` с типом `note`. Для нее используются поля `title`, `content`, `folderId`, `status`, даты создания и обновления. Создание заметки возможно как через страницу заметок, так и через чат-команду, начинающуюся с ключевого слова. Старый режим автоматического сохранения произвольных фраз как заметок отключен через константу `LEGACY_AUTO_SAVE_ENABLED = false`, поэтому обычный вопрос пользователя не превращается в запись без явного намерения.

Папки. Папки хранятся в таблице `folders`. При регистрации пользователя backend создает набор системных папок. Пользовательские папки можно создавать, переименовывать, менять им цвет и иконку, а также удалять. Системные папки защищены: маршруты изменения и удаления проверяют поле `isSystem` и не позволяют выполнять эти операции над системными папками. При удалении обычной папки объекты не теряются, а остаются без папки.

Файлы. Файл является объектом `items` с типом `file`. Для него дополнительно сохраняются `originalFilename`, `mimeType`, `fileSize` и `fileData`. Файл может быть привязан к папке, отображаться на странице файлов или внутри конкретной папки. Пользователь может задавать ассистенту вопрос по файлу, если файл выбран в интерфейсе.

Напоминания. Напоминание является объектом `items` с типом `reminder`. Основное отличие от заметки - наличие поля `reminderAt`, в котором хранится дата и время напоминания. Статус `completed` используется для отметки выполненного напоминания. В текущей версии напоминания отображаются в приложении и участвуют в ближайших событиях, но push-уведомления и фоновые системные уведомления не реализованы.

Списки. Список является объектом `items` с типом `list`. Его пункты хранятся в поле `content` в виде JSON-структуры. На frontend список отображается как набор пунктов с чекбоксами. Изменение состояния пункта приводит к обновлению объекта через API. Такой вариант проще отдельной таблицы пунктов списка и подходит для текущего масштаба приложения.

Поиск. Поиск реализован через endpoint `/api/search`. Он использует текстовое сопоставление по заголовку, содержимому, краткому описанию и оригинальному имени файла. В коде используется `ilike`, поэтому поиск является обычным текстовым поиском по строкам. Семантический поиск, embeddings и векторное ранжирование в текущей версии отсутствуют.

## 3.8. Сборка и запуск проекта

Проект можно запускать локально через пакетный менеджер `pnpm` или через Docker Compose. В корневом `package.json` определены общие команды `build`, `typecheck` и `typecheck:libs`. В backend-приложении доступны команды `dev`, `build`, `start`, `typecheck`, `test` и `test:commands`. Во frontend-приложении доступны команды `dev`, `build`, `serve` и `typecheck`.

Для контейнерного запуска используется файл `docker-compose.yml`. Он описывает три сервиса:

- `db` - PostgreSQL 16 с базой `mindvault`;
- `api` - backend-сервер на порту `8080`;
- `web` - frontend-приложение на порту `18174`.

Сервис `api` зависит от готовности базы данных и перед запуском применяет схему командой `pnpm --filter @workspace/db run push`. Затем запускается backend в режиме разработки. Сервис `web` запускает Vite dev server и проксирует API-запросы на backend-сервис внутри Docker-сети.

| LLM-классификация команд не используется как основной механизм | В маршруте классификации применяется серверная функция обработки действий, а AI-код ниже возврата не участвует в текущем потоке | Либо удалить недостижимый код, либо переработать классификацию намерений с явным тестированием |

Перечисленные ограничения не снижают ценность текущей реализации для целей ВКР, но задают честные границы описания. В основной части работы следует писать о том, что подтверждено кодом: веб-интерфейс, серверное API, PostgreSQL-схема, авторизация, CRUD-операции, загрузка файлов, чат, контекст ассистента, fallback при недоступности AI и Docker Compose для локального запуска. Функции, отсутствующие в коде, должны оставаться в разделе перспектив развития.

## Выводы по главе 3

В третьей главе рассмотрена практическая реализация веб-приложения MindVault. Проект выполнен как TypeScript-монорепозиторий, включающий frontend на React и Vite, backend на Express, схему PostgreSQL на Drizzle ORM, сгенерированные API-клиенты и интеграцию с OpenAI/OpenRouter-совместимым API.

На frontend реализованы защищенные маршруты, страницы авторизации, чата, заметок, файлов, напоминаний, списков, папок и настроек. На backend реализованы маршруты авторизации, работы с папками, универсальными объектами, поиском, статистикой, файлами и сообщениями чата. Данные пользователей изолируются по `userId`, а доступ к защищенным маршрутам выполняется через JWT.

База данных построена вокруг таблиц `users`, `folders`, `items`, `conversations` и `messages`. Универсальная таблица `items` используется для хранения заметок, файлов, напоминаний и списков, что упрощает общий поиск и формирование контекста для ассистента. Файлы сохраняются с метаданными и base64-содержимым, а поддерживаемые форматы проходят извлечение текста.

ИИ-ассистент реализует два типа поведения: ответы на сообщения с использованием пользовательского контекста и выполнение прикладных команд через серверную логику. При недоступности внешнего AI-провайдера предусмотрен fallback-ответ. При этом функции, не подтвержденные кодом, такие как push-уведомления, OCR, векторный поиск, production-деплой, совместная работа и end-to-end encryption, отнесены к ограничениям и перспективам развития.

Проведенная проверка показала, что автоматические тесты командного парсера backend выполняются успешно, а TypeScript-проверка frontend и backend не выявляет ошибок. Для финальной версии ВКР необходимо дополнить главу реальными скриншотами интерфейса, результатами Docker-запуска и скриншотом выполнения тестов.
